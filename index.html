Architectural Notes & Open Questions

Overall Assessment

SOLID The structure is architecturally sound. The App Token / User Token separation is the standard OAuth2 pattern for B2B+B2C integrations of this type. Phase 3 positioning (first login after card receipt) reflects the realistic onboarding sequence.

A note on sequentiality

The diagram represents the logical flow, not a strict temporal sequence. In practice, Phase 1 (App Token) runs once at HR platform bootstrap and is reused across many sessions — it is not re-triggered before each Phase 2 call. Similarly, Phase 3 (User Token) is cached per session. The arrows show causal dependency, not real-time ordering.

Open Questions — to verify with Pellegrini

UNCONFIRMED Token type for AttivaCarta. Assigned here to the User Token flow (Phase 3), which is logical — the user activates their own card. However, this is not explicitly stated in the v1.0 doc. If it requires an App Token, Phase 3 would need to be repositioned.

UNCONFIRMED Pre-existing Pellegrini accounts. Phase 3 assumes the user's first login happens after card receipt. If a user already has a Pellegrini account (e.g., from a previous employer), the onboarding flow bifurcates and this sequence may not apply.

UNCONFIRMED Token refresh strategy. Both App Token and User Token are shown as "cached for session". Expiry and refresh logic are not documented in v1.0. For production integrations, silent refresh or re-authentication windows need to be defined.

MISSING Auth flow for institutional partners. The doc lists Auth Code and ROPC as alternatives for User Token, but does not specify which applies to B2B integrations. Auth Code requires a browser redirect; ROPC is deprecated (RFC 9700). Critical architectural decision — must be confirmed before any integration work begins.

What Pellegrini should provide

For a B2B API of this complexity — two distinct OAuth flows, endpoints with different token requirements, non-trivial onboarding sequence — the expected documentation is: a narrative integration guide, a reference sequence diagram and possibly a Postman collection. This flow was reconstructed from endpoint naming and business logic alone, which means the open questions above require direct confirmation from Pellegrini's technical team before any integration assumptions can be treated as final.
