This document collects all open questions arising from the preliminary analysis of the Pellegrini API v1.0, the Bible Architectural Analysis, and the Integration Architecture Report. It is structured in three sections: External questions to be submitted to the Pellegrini / DIT team, Engineering questions for internal technical alignment, and Commercial & Product questions for business and go-to-market decisions. Questions are ordered by priority within each subsection. None of the architectural or commercial decisions described here are final — this document is a working input for the evaluation phase.

External
Section 1 — Questions for Pellegrini / DIT
These questions must be formally submitted to the DIT / Pellegrini technical team before any integration work can begin. Several are hard blockers — highlighted in the notes.
API & Technical Gaps
Q01CriticalWhat endpoint does the POS terminal call to validate a CodiceSpesa? What is the full request/response schema?
The POS-side validation flow is completely absent from Pellegrini API v1.0 documentation — without this, the end-to-end payment flow cannot be fully designed.
Expected details: endpoint name, HTTP method, auth mechanism, error codes, timeout behaviour.
Q02CriticalIs there a mechanism to credit / load BPE vouchers into a user's Pellegrini wallet from an external platform?
Fundamental integration prerequisite: Coverflex holds the user BPE budget — how does it transfer or fund the Pellegrini-side balance?
  • Is there an endpoint such as /AccreditaBuoni, /CaricaSaldo, or similar?
  • Or is the budget loaded at the time of order creation (CreaOrdine)?
  • Or is there a batch / financial reconciliation process entirely separate from this API?
Q03HighDoes the Pellegrini API support webhooks or push notifications to inform third-party backends when a POS transaction completes?
Without webhooks, third-party integrations must poll GetMovimentiCarta — which introduces latency and is inefficient at scale.
If webhooks exist: what events are emitted, what is the payload schema, and how is the endpoint registered?
Q04HighWhich endpoints require a User Token, and which can operate with an App Token (client_credentials)? Specifically: can POST /SpendiBPE be called with an App Token?New
The API documentation does not explicitly classify endpoints as AppOnly or UserOrApp. The token type required for SpendiBPE has major architectural implications and must be clarified before integration design can begin.
Q05HighFor partner / institutional integrations, what OAuth2 flow does Pellegrini recommend to obtain a User Token? Is there a dedicated flow that does not require handling individual user credentials?New
The API documentation (§1.3) mentions Authorization Code or ROPC as options, without specifying which applies to partner scenarios. The choice of flow has direct implications for security posture, UX model, and credential handling — this needs to be explicitly scoped with Pellegrini / DIT.
Q06HighIn POST /SpendiBPE, is 'Pin' the physical card PIN or an app-level PIN set by the user inside the Pellegrini app? Who sets it for virtual / digital badges?
The documentation states '$$$Pin' format — the triple-dollar prefix is never explained anywhere.
This affects UX design significantly: card PIN requires the user to remember a banking credential; app PIN is a simpler code set at onboarding.
Q07HighWhat is the User Token expiry? Is there a token refresh or silent re-authentication mechanism documented?
Token expiry is not documented. No token refresh or silent re-authentication flow is described in the API documentation — this affects session management design.
Q08HighFor employees with isFullDigital: true badges — does the onboarding flow skip physical card issuance (CreaOrdine, AttivaCarta) entirely?
If virtual badges are available without a physical card, this significantly simplifies the onboarding path.
Specifically: does a fully digital badge still require Pellegrini-side BPE balance to exist before spend, or does it work differently?
Q09MediumWhy is CodiceSpesa typed as 'decimal' in the /SpendiBPE response? Is it safe to treat it as integer or string?
A spend code used for POS validation should logically be an integer or string — decimal typing is anomalous.
Does the spend code ever contain decimal places? Or is 'decimal' just a documentation choice?
Q10MediumIs the 'Badge' field truly mandatory in POST /SpendiBPE, or can Pellegrini resolve it internally from 'Mail'?
GetSaldo already resolves Mail → Badge server-side. If Badge can be omitted in SpendiBPE, the client needs to cache less data.
Q11MediumWhat is the 'Provenienza' field in POST /SpendiBPE? What are the valid values and when is it required?
This field appears in payload examples but is absent from the parameter table — its purpose is undocumented.
Observed value in examples: "APP". Are other values valid (e.g. "WEB", "POS")?
Q12MediumHow can a caller distinguish 'card just created' from 'card already exists' in the POST /VerificaERichiediCarta response?
Both cases currently return the identical message 'Carta Richiesta' — there is no differentiation in the documented response.
Q13MediumHow long after POST /VerificaERichiediCarta does the badge become available and usable via the API?
Timing is undocumented. A third-party integrator needs to know whether to poll, wait a fixed interval, or use a different signal to confirm badge readiness.
Q14MediumCan a user have multiple active badges simultaneously? If so, how should an integrating application decide which badge to use in POST /SpendiBPE?
Not addressed in the documentation — relevant for users who may have both physical and digital badges active.
Q15MediumWhat are endpoints TrackingOrdine, TrackingSpedizioni, and logBadge? When will they be documented?
Three of the endpoints listed in the API have completely empty sections in Pellegrini API v1.0 documentation.
Q16MediumIs there a complete API error code catalog with descriptions and recommended handling?
The documentation does not include any error codes or error response schemas.
Expected: HTTP status codes, application-level error codes, descriptions, and retry guidance.
Integration Model & Partnership
Q17HighIs Pellegrini API v1.0 intended for direct partner integration, or does a separate B2B / partner integration guide exist alongside the consumer-facing API?
The current documentation appears to describe the API as used by Pellegrini's native app. It is unclear whether the same API surface, authentication flows, and data models are appropriate for institutional partner use cases — or whether a separate specification exists.
Q18HighIs there a sandbox / staging environment available for integration testing?
No sandbox is mentioned anywhere in the documentation. End-to-end testing without one requires either a mock or a limited live pilot — both with significant operational risk.
Q19HighCan you provide the Azure B2C Client ID and Client Secret (currently redacted in the documentation) for sandbox / test environment setup?
Without these credentials it is impossible to obtain any token, test any endpoint, or begin development — this is a prerequisite for any technical work.
Q20MediumAre there rate limits, throttling policies, or usage quotas on the Pellegrini API?
Not documented. Relevant for polling strategies and batch operations at scale.
Q21MediumWhat are the SLA terms — uptime guarantee, planned maintenance windows, and incident response time?
Q22MediumWho is the technical point of contact for integration support and issue escalation?
Q23MediumWhat is the data ownership model? Under GDPR, who is the data controller for user personal data stored in the Pellegrini system?
Pellegrini stores email, fiscal code, badge data, and transaction history for Coverflex users. The controller/processor relationship and data retention policies need to be formally defined.

Engineering
Section 2 — Internal Engineering Questions
These are design and feasibility questions for the Coverflex engineering team. Some depend on answers from Pellegrini — flagged in the context notes.
Architecture Decisions
Q25HighShould Coverflex act as a full proxy (APP → Coverflex Backend → Pellegrini) or allow the APP to call Pellegrini directly for certain flows?
A full proxy (all calls routed through the Coverflex Backend) centralises auth and logging but adds latency. Direct APP-to-Pellegrini calls reduce hops but expose Pellegrini tokens to the client. The tradeoffs need to be explicitly scoped — particularly in light of the open questions around token type and credential handling (Q04, Q05).
Q26HighWhen and how do we sync / credit the user's BPE budget from Coverflex to Pellegrini's servers?
This is the most critical architectural unknown — depends on the answer to Q02. Options to evaluate:
  • Batch top-up: Coverflex credits Pellegrini at the start of each benefit period
  • On-demand: Coverflex credits Pellegrini the moment the user tries to spend
  • Order-time: CreaOrdine already provisions the budget (needs DIT confirmation)
Q27HighHow do we handle the auth flow and credential / token lifecycle for obtaining User Tokens on behalf of Pellegrini users?
Depends heavily on the answer to Q04 and Q05. The implications of each possible flow (Auth Code, ROPC, or App Token if allowed) span security posture, UX model at spend time, token storage, refresh strategy, and session management. These tradeoffs need to be explicitly mapped before any implementation decision is made.
Q28MediumWhere do we store PomId and Badge for each user — resolved once at onboarding and persisted, or fetched fresh on each session?
PomId is stable per employee. Badge can expire (dataScadenzaBancaria). Caching strategy must account for badge expiry and renewal.
Q29MediumHow do we handle the Pellegrini App Token (client_credentials) lifecycle — generate per-request, cache with TTL, or long-lived?
Token expires in 3600s. A token manager service on the Coverflex Backend should handle renewal transparently — approach and failure handling need to be designed.
Q30MediumWithout a Pellegrini webhook, what is our polling strategy for post-spend balance updates?
Options to evaluate:
  • Poll immediately after spend code generation
  • Poll on next app open
  • Push a local optimistic update and reconcile later
Q31MediumHow do we handle concurrent spend code generation — e.g. user opens the app on two devices simultaneously?
Pellegrini may or may not prevent duplicate active codes. A locking or deduplication strategy on the Coverflex side needs to be defined.
Security & Data
Q32HighIf User Tokens are involved in the integration, how do we prevent them from being exposed to the client-side app?
Depends on the resolution of Q04/Q05. The token handling strategy — where tokens are stored, how they transit, and who can access them — needs to be explicitly scoped as part of the security design.
Q33HighHow do we handle the PIN in the SpendiBPE flow — does it transit through Coverflex Backend or is it sent directly from the APP to Pellegrini?
The two paths have different security profiles. Transmitting a PIN through a proxy adds intermediary exposure risk; a direct app-to-Pellegrini call reduces it but may conflict with the proxy architecture. The choice has implications for security responsibility and integration model.
Q34MediumWhat data minimisation policy do we apply to Pellegrini user data (fiscal code, badge number, transaction history) stored on Coverflex servers?
Q35MediumHow do we handle GDPR right-to-erasure requests for users who have data in both Coverflex and Pellegrini systems?
Implementation & Testing
Q36HighWithout a Pellegrini sandbox environment, how do we test the integration before going live?
Options to evaluate: request a sandbox from DIT, build a Pellegrini API mock based on the documentation, or negotiate a limited live test with a pilot employer.
Q37MediumHow do we handle the three undocumented endpoints (TrackingOrdine, TrackingSpedizioni, logBadge) — skip, stub, or block on DIT to provide documentation?
Q38MediumWhat is our error handling strategy for Pellegrini API failures mid-spend-flow?
e.g. SpendiBPE succeeds but the 120s window expires before POS confirmation — who resolves the inconsistency, and what is the user-facing response?
Q39MediumDo we need to build a Pellegrini balance reconciliation job to ensure Coverflex and Pellegrini balances stay in sync?
If balances can drift (e.g. due to a failed sync or a missed transaction event), a reconciliation process is needed to detect and resolve discrepancies.
Q40MediumWhat observability do we need on Pellegrini API calls — logging, tracing, alerting on error rates?
Given Pellegrini API is a third-party dependency with unknown reliability, circuit breakers and fallback UX strategies for when it is unavailable need to be scoped.
Commercial · Product
Section 3 — Commercial & Product Questions
These are for the commercial, product, and go-to-market teams. They define constraints and assumptions that directly influence the technical design.
Business Model & Feasibility
Q41HighIs Coverflex willing to issue physical Pellegrini cards to Coverflex users, or must we pursue a fully digital (isFullDigital) path?
Physical card issuance adds onboarding friction, shipping lead time, and cost. isFullDigital may not be available for all use cases — needs DIT confirmation (Q08).
Q42HighHow does the parallel BPE ledger work — if Pellegrini holds the operational balance, does Coverflex maintain a shadow ledger?
If the user's Pellegrini balance drifts from Coverflex's record (e.g. a failed sync), who is the source of truth and what is the user-facing resolution?
Q43HighWhat are the commercial terms with Pellegrini — does Coverflex pay a per-user fee, a per-transaction fee, or a fixed annual fee?
Q44MediumWhich employers are in scope for the first pilot? What is the minimum viable rollout size to validate the integration?
Q45MediumAre there Italian welfare regulation constraints (MIMIT authorisation, Law 193/2024) that affect how BPE vouchers are handled when routed through a third-party platform like Coverflex?
Fringe benefit tax treatment and compliance requirements may differ when an intermediary is involved. Law 193/2024 caps BPE commissions at 5% — the implications for a dual-operator chain need to be assessed.
Q46MediumWhat happens to the user's Pellegrini wallet when they leave a company — is the balance managed and returned by Coverflex, or by Pellegrini directly?
Product & UX
Q47HighWhat is the acceptable onboarding flow length for a Coverflex user activating Pellegrini benefits?
The current documented onboarding involves card issuance, shipping, and physical activation — potentially weeks of friction before first use. A fully digital path needs to be confirmed as viable.
Q48HighHow do we communicate the 120-second spend code window to the user in the Coverflex app UX?
120 seconds is tight — especially if the POS queue is long. Is there a way to regenerate codes rapidly if the first expires? Can the code be displayed as a QR or barcode, or only as digits?
Q49MediumHow do we communicate the merchant network coverage difference to users — Coverflex today covers any Visa POS, Pellegrini covers only contracted merchants?
For users who currently use BPE at non-contracted merchants (via the Coverflex Visa card), integration with Pellegrini's network could represent a coverage reduction. This needs to be surfaced and scoped before rollout.
Q50MediumDo we need to educate Coverflex users on the split payment model — that only the voucher amount is paid via Pellegrini, with the remainder by card / cash?
Q51MediumIf a spend code fails at the POS (expired, validation error, network issue) — who handles user support, Coverflex or Pellegrini?
Clear responsibility assignment is needed before launch — user-facing failures at the POS are high-stress situations requiring a defined escalation path.
