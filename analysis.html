🚨 Strategic Note — Before Proceeding

This is not just a technical integration. It is a strategic dependency on a direct competitor: user identity, balance, and transaction data would reside on Pellegrini's servers. Alternative architectures (BIN acceptance, dedicated B2B API) may preserve data sovereignty and UX quality — and should be assessed before committing to the API path.

Section 1 — The Two Architectures: A Fundamental Conflict

🏢 Coverflex — Card-Based Model (ISO 8583 / Visa)
User taps Coverflex Visa card (physical or mobile wallet) at any POS
POS generates ISO 8583 authorization request → Visa Network → Paymentology
Coverflex Backend validates merchant, applies split algorithm: BP → Fringe → Wallet Credit
ZERO user interaction — no app, no PIN, no code generation
Works at ANY Visa-enabled POS, including offline (Visa offline mode)
Multi-wallet automatic split — BP + Fringe Benefits + Wallet Credit in one tap
✓ ~2 seconds. User experience transparent and frictionless.
VS
🌐 Pellegrini — Token-Based Model (OTP / CodiceSpesa)
User opens Pellegrini app (or Coverflex app with integration)
Authenticates to Azure AD B2C → User Token (Auth Code or ROPC — see §2)
Selects vouchers, enters PIN in format $$Pin (undocumented)
Calls POST /SpendiBPE → receives CodiceSpesa (decimal, ~120s validity)
User shows or dictates 6-digit code to cashier
Cashier manually types code into POS; POS validates against Pellegrini system
Single-wallet BPE only — no multi-wallet split, no fallback to other wallets
✗ ~15–30 seconds, requires network, PIN, active user interaction. Fails without internet.

Section 2 — Authentication: User Token & Credential Management Problem

📋 What the Pellegrini API Doc States

Section 1.3: "Per chiamare gli endpoint UserOrApp come utente, il token deve essere ottenuto tramite un flusso utente B2C (ad esempio Authorization Code oppure ROPC, se previsto)". The doc lists both flows as options; it does not mandate one over the other. Additionally, the API does not explicitly classify which endpoints require a User Token vs an App Token — each endpoint's token requirement is undocumented.

Points Requiring Attention

⚠ ROPC is a Deprecated OAuth2 Flow

ROPC (Resource Owner Password Credentials) is deprecated per RFC 9700 and Microsoft's own guidance. If the chosen integration design uses ROPC, this warrants explicit evaluation of security posture and alternatives.

⚠ Token Acquisition & Credential Handling

The choice of auth flow (Auth Code vs ROPC) has direct implications for how and where user credentials are handled, the UX model at spend time, token lifecycle management, and security responsibility. These tradeoffs need to be explicitly scoped during integration design.

⚠ Token Scope per Endpoint — Undocumented

It is unclear whether SpendiBPE requires a User Token or can operate with an App Token. This has major architectural implications and must be clarified with Pellegrini before design begins.

⚠ PIN Setup Gap

At no point in the documented API is there a PIN creation or reset endpoint. If the user doesn't already have a Pellegrini PIN, the entire spend flow is blocked. Coverflex has no mechanism to set or reset a Pellegrini PIN on behalf of a user.

Section 4 — The Balance Problem (Fundamental Blocker)

🔴 Critical Blocker — Balance Residency Conflict

POST /SpendiBPE requires a ListaBuoni — a list of actual vouchers that must physically exist on Pellegrini's servers for that user's Badge. Coverflex holds the user's BPE budget internally. Pellegrini has zero knowledge of it. Without pre-loading vouchers onto Pellegrini, SpendiBPE cannot be called and no CodiceSpesa can be generated. CodiceSpesa is not just an auth code — it is the materialisation of a verified Pellegrini-side balance. No Pellegrini balance = no code, regardless of what Coverflex holds. This is a fundamental architectural blocker of the integration.

⚠ How to Load Balance onto Pellegrini — No Documented Endpoint

The Pellegrini API has no documented endpoint to credit vouchers from an external platform. Three possible mechanisms — all unconfirmed — must be clarified with DIT:

Option A — Periodic Batch Load

Coverflex credits Pellegrini at start of each benefit period (monthly). Coverflex pre-pays Pellegrini upfront and becomes a voucher distributor. Manages two ledgers. Business model shift.

Option B — On-Demand Load

Coverflex credits Pellegrini in real-time when user opens the spend screen. Requires an undocumented top-up API. Real-time latency and consistency risks.

Option C — Order-Time Load

CreaOrdine might implicitly provision vouchers. Coverflex triggers orders when users earn BPE budget. Needs DIT confirmation on how CreaOrdine relates to balance.

Section 5 — Payment Flow at the POS

ℹ How Pellegrini Payment Actually Works

The Pellegrini spend flow is an active, multi-step user process — not a transparent card payment. The user must consciously initiate, authenticate, and present a temporary code. This is the dominant pre-digitalization BPE model in Italy — fundamentally different from Coverflex's invisible card-based approach.

User-Side Spend Flow

  1. Cashier announces "Buoni Pasto" at checkout — user declares intention to pay with vouchers
  2. User opens Coverflex app → navigates to Pellegrini BPE spend section
  3. Coverflex Backend calls POST /GetSaldo with Mail + Badge + IdDispositivo → retrieves live balance and voucher list (DettaglioSaldo)
  4. User sees balance + expiring vouchers, selects amount and vouchers to use (ListaBuoni selection)
  5. User enters Pellegrini PIN (format $$Pin — undocumented; likely physical card PIN or app PIN)
    Is this the physical card PIN or an app-level PIN set by the user? Who sets it for virtual badges?
  6. Coverflex Backend calls POST /SpendiBPE with Mail + Badge + PomId + ListaBuoni + Pin + IdDispositivo
    Endpoint to get IdDispositivo (device_id) tbd
  7. Pellegrini validates balance → returns CodiceSpesa (Decimal type, ~6 digits) + SecondiRimanenti (120s) + Valore
  8. App displays CodiceSpesa prominently with 120-second countdown timer
    Can the code be displayed as QR or barcode, or only as digits?
  9. User shows or dictates the code to the cashier

POS-Side & Confirmation Flow

  1. Cashier types CodiceSpesa into POS terminal manually
    Does any POS support QR/barcode scanning of the code? Not documented.
  2. POS calls a Pellegrini endpoint to validate the code
    Which endpoint does the POS call? Entirely absent from Pellegrini API documentation
  3. Pellegrini validates code, deducts vouchers from user's Pellegrini balance, confirms amount to POS
  4. POS shows receipt: voucher portion settled, remaining amount paid by card/cash
  5. Coverflex Backend polls POST /GetMovimentiCarta to detect transaction confirmation (no webhook available)
    How does Pellegrini notify Coverflex when a transaction completes and what's its payload? No webhook is documented.
  6. Coverflex marks transaction as USED, updates displayed balance in app
    How does Pellegrini notify Coverflex for reconciliation?
⚠ No Transactional Push Confirmation

There is NO documented webhook or callback. After generating CodiceSpesa, Coverflex does not know in real time whether the code was actually used. The only mechanism is polling GetMovimentiCarta — creating state ambiguity and customer support issues. A code could expire without being used, or be used without Coverflex being notified promptly.

CodiceSpesa State Machine

PENDING
→ SpendiBPE called, awaiting response
ACTIVE
→ CodiceSpesa received, visible to user, countdown running
EXPIRED
→ SecondiRimanenti = 0, code not used
UNKNOWN
→ Code expired but no movement found yet — ambiguous state
USED
→ Polling finds matching movement in GetMovimentiCarta
CANCELLED
→ AnnullaCodiceSpesa called before expiry
NOT_USED
→ Polling after 5 min timeout finds no movement

The UNKNOWN → USED/NOT_USED transition depends entirely on polling — an unreliable and latent confirmation mechanism. No refund is available if BP transaction is already processed.

Section 6 — The Model Conflict

ℹ See §1 for the full architectural and UX comparison

§1 documents the side-by-side breakdown across all dimensions — payment paradigm, UX, multi-wallet, acceptance network, offline support, and data sovereignty. The key structural consequence that flows from those differences is captured below.

💡 Consequence

To use the Pellegrini API as-is, Coverflex would need to pre-load vouchers on Pellegrini for every user — paying upfront, managing reconciliation across two ledgers, navigating an unconfirmed auth flow, and potentially managing Pellegrini user tokens on behalf of each user. The business model and data sovereignty implications of this must be weighed explicitly.

Section 8 — Architectural Options

Network Interoperability — BIN Acceptance
Pellegrini configures its POS network to accept the Coverflex Visa Debit BIN as a valid BPE payment. User pays with their existing Coverflex card (tap/swipe). Coverflex authorises BPE in real-time via ISO 8583. No Pellegrini API needed. No Badge. No PIN. No code generation. No credential management.
Best UX — user pays with existing Coverflex card. Zero extra steps.
No balance duplication — Coverflex remains authoritative balance holder
No data sovereignty issue — no user data sent to Pellegrini
No ROPC/credential management problem
Requires commercial + network agreement between Pellegrini, Coverflex, and Visa/card network
Not an API integration — outside the scope of the current Pellegrini API documentation
Timeline unpredictable — depends on Pellegrini's willingness and network config
Is Pellegrini open to accepting external BINs on their POS network?
Option B · Needs investigation
Dedicated B2B Partner API (Backend-to-Backend)
Pellegrini exposes a separate B2B API that Coverflex calls as an institutional partner — without individual user Badges or ROPC. Coverflex manages user balance internally; the partner API handles POS-side authorisation on Coverflex's behalf. This API is not documented in current Pellegrini API v1.0.
Clean architecture — each system keeps its own balance
No ROPC / no credential management problem
No physical or virtual card issuance required
May not exist — needs explicit DIT confirmation
